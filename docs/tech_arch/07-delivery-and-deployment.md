## 7. 开发阶段建议

## 7.1 里程碑划分

### Milestone 1：基础房间与连接能力

目标：

- 创建房间
- 分享房间码
- 加入房间
- WebSocket 联机
- 玩家重连
- Lobby 状态同步

优先级最高，因为这是后续一切的运行基础。

### Milestone 2：核心引擎最小闭环

目标：

- 角色选择
- 场景选择
- 初始板块
- 回合流转
- 移动
- 发现新房间
- 抽 Event / Item / Omen
- 属性变化
- haunt roll

这个阶段先做完整的“作祟前可玩闭环”。

### Milestone 3：作祟后通用战斗与隐藏信息

目标：

- 作祟触发
- 阵营切换
- 视角隔离
- 攻击与伤害
- 怪物基础规则
- 死亡与尸体搜刮

这个阶段完成“引擎层面”的作祟后支持。

### Milestone 4：内容系统与样板作祟

目标：

- 静态内容加载
- 1 个普通奸徒作祟
- 1 个无奸徒作祟
- 1 个隐藏奸徒或自由混战作祟

这是验证架构是否真的能承载内容分叉的关键阶段。

### Milestone 5：完整内容扩充

目标：

- 全部角色
- 全部房间
- 全部卡牌
- 全部场景
- 全部 50 个作祟

### Milestone 6：体验与运维补强

目标：

- 动画与交互优化
- PWA 安装体验
- 完整重连恢复
- 回放/审计工具
- 指标与日志完善

## 7.2 MVP 最小功能集

MVP 不应追求“全部 50 个作祟全部上线”，而应追求“证明整条技术链是对的”。

推荐 MVP 包含：

- 房间创建、分享、加入、断线重连
- 3-6 人 lobby
- 角色选择与场景选择
- 作祟前完整流程：
  - 移动
  - 探索
  - 房间放置
  - 三类抽牌
  - 属性变化
  - haunt roll
- 作祟后通用引擎：
  - 阵营
  - 隐藏信息过滤
  - 攻击
  - 伤害
  - 怪物基础规则
  - 尸体与拾取
- 至少 2-3 个代表性作祟
- 桌面端与移动端都可完整打一局

MVP 不必包含：

- 全部 50 个作祟
- 账号体系
- 排位或匹配
- 观战模式
- 聊天系统
- 回放 UI

## 7.3 开发优先级建议

推荐优先级：

1. 房间与实时连接
2. 权威状态机与命令/事件模型
3. 地图与板块放置
4. 牌堆与属性系统
5. 作祟触发与视角过滤
6. 战斗、怪物、死亡
7. 内容配置化
8. 动画、PWA、运维增强

原因：

- 前 5 项决定架构是否成立
- 后 3 项是在架构成立后的规模化与体验提升

## 8. 推荐的 docker compose 结构

首版建议的 compose 服务：

- `web`
  - React 构建产物
  - 由 Caddy 提供静态服务
- `api`
  - Go 单体服务
- `db`
  - PostgreSQL
- `migrate`
  - 一次性执行数据库迁移

推荐关系：

- `web` 反向代理 `/api` 和 `/ws` 到 `api`
- `api` 连接 `db`
- `migrate` 在 `api` 启动前跑完 schema

这样就已经足够支撑首版部署与本地开发。

## 9. 最终建议

这款游戏最正确的首版做法不是拆更多服务，而是把这三个底座做扎实：

1. **房间级串行运行时**
2. **权威状态 + 操作日志 + 状态快照**
3. **按玩家视角投影的隐藏信息模型**

只要这三点成立：

- 地图增长能做对
- 作祟分叉能做对
- 断线重连能做对
- 50 个剧本也有可持续的承载方式

对《山屋惊魂》这类规则密度极高、信息不对称明显、联机人数较小的桌游来说，这比任何“服务拆分式复杂架构”都更贴合实际。
