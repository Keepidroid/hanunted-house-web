> 目标：统一“单体 Go 服务 + 房间级运行时 + 权威状态机”这一整体架构认知。

## 0. 架构结论先行

本项目推荐采用：

- **单体 Go 实时服务**
- **房间级内存态运行时**
- **PostgreSQL 持久化**
- **WebSocket 负责房间内实时同步**
- **REST 负责房间生命周期和初始加载**
- **操作日志 + 状态快照** 作为恢复、审计、回放与排障基础

这是比“大厅服 / 对局服 / 内容服”更合适的首版方案，原因很直接：

- 游戏规模小，一局仅 3-6 人。
- 实时交互频率不高，但规则一致性要求极高。
- 真正复杂的是**规则引擎、地图增长、作祟分叉、信息不对称**，不是服务拆分。
- 单体房间模型最容易保证“同一房间内所有操作串行、状态绝对一致”。

因此，系统的核心设计重点不是“怎么拆服务”，而是：

1. 如何把一局游戏建模成可验证、可恢复、可过滤视角的权威状态。
2. 如何让每个房间成为一个串行处理命令的状态机。
3. 如何支持作祟后的隐藏信息和剧本覆写。
4. 如何在断线、重连、刷新、服务重启后恢复到正确视图。

## 1. 整体架构概览

### 1.1 系统分层与模块划分

推荐采用 4 层结构。

#### 第一层：客户端层

- React Web 客户端
- PC 浏览器与移动端浏览器共用同一套前端
- 支持 PWA 安装、静态资源缓存、断线提示和重连

职责：

- 渲染房间、地图、角色、手牌、弹窗、动画
- 收集玩家操作并发送给服务端
- 保存当前玩家的会话令牌、重连信息、UI 状态
- 渲染经过权限过滤后的玩家视角状态

#### 第二层：接入层

- HTTP API
- WebSocket Gateway

职责：

- 处理创建房间、加入房间、恢复会话、获取初始快照等 REST 请求
- 管理 WebSocket 连接、心跳、重连、房间订阅
- 将客户端命令转发给对应 `RoomRuntime`

#### 第三层：房间运行时层

这是项目的核心。

- `RoomManager`
- `RoomRuntime`
- `GameEngine`
- `ViewProjector`
- `PersistenceWriter`

职责：

- 每个房间维护一个权威内存态
- 串行处理该房间中的所有命令
- 执行规则校验、随机结算、回合推进、作祟切换
- 生成公开/私密事件
- 为每个玩家投影出可见视图
- 写入操作日志和快照

#### 第四层：持久化与内容层

- PostgreSQL
- 静态内容定义文件（角色、房间、卡牌、作祟）

职责：

- 持久化房间元数据、玩家座位、会话令牌、操作日志、快照
- 存储可审计的历史记录
- 提供服务重启后的恢复基础
- 以版本化静态内容驱动规则引擎

### 1.2 前后端职责边界

推荐采用**服务端权威裁定，前端纯视图与交互层**的边界。

#### 前端负责

- 用户输入与操作意图表达
- 地图、角色、卡牌、动画、日志的可视化
- 局部 UI 状态，如当前选中房间、打开的弹窗、拖拽/缩放、Hover 提示
- 弱交互预校验，例如按钮是否可点击、可选目标高亮

#### 后端负责

- 真正的规则合法性判定
- 回合和阶段推进
- 随机结果生成
- 房间牌堆、板块堆、属性变化、伤害、作祟触发、胜负判定
- 隐藏信息隔离
- 断线重连后的正确视图恢复

原则：

- **客户端永远不能自己算出并提交“结果”，只能提交“意图”。**
- 例如前端提交“我要从 A 走到 B 并探索该门”，最终是否合法、抽到什么、是否作祟、作祟后谁可见什么，都由后端裁定。

### 1.3 整体数据流与交互流程

#### 流程 A：创建房间与加入房间

1. 玩家打开首页，输入昵称。
2. 前端调用 `POST /api/rooms` 创建房间。
3. 后端创建 `room`、`seat`、`player_session`，返回房间码与玩家令牌。
4. 房主分享房间链接给其他玩家。
5. 其他玩家调用 `POST /api/rooms/{code}/join` 加入。
6. 前端建立 WebSocket 连接并订阅该房间。
7. 服务端返回当前 lobby 视图。

#### 流程 B：房间内操作

1. 玩家点击一个动作，例如“移动到房间”“抽牌选择”“执行特殊行动”。
2. 前端通过 WebSocket 发送 `command`。
3. 该命令进入对应房间的 `RoomRuntime` 邮箱。
4. `RoomRuntime` 校验当前玩家、当前阶段、资源条件、目标合法性。
5. `GameEngine` 产生一个或多个领域事件并应用到权威状态。
6. `PersistenceWriter` 写入事件日志，并按策略刷新快照。
7. `ViewProjector` 为每个在线玩家生成过滤后的视图更新。
8. WebSocket 广播公开事件，并向不同玩家发送不同的私密数据。

#### 流程 C：重连

1. 浏览器刷新或断线后重新打开页面。
2. 前端携带本地保存的 `playerToken` 调用恢复接口或直接重新连接 WS。
3. 后端校验令牌归属座位。
4. 若房间仍存在，则返回该玩家当前可见的完整快照。
5. 前端以该快照重建本地状态，并继续接收增量事件。

#### 流程 D：服务重启恢复

1. 服务启动时不必立即恢复所有房间。
2. 当某个活动房间有玩家重新连接时，服务从 PostgreSQL 读取：
   - 房间元数据
   - 最新快照
   - 快照后的事件尾部
3. 重建 `RoomRuntime` 的权威状态。
4. 重新投影各玩家视图。

这个按需恢复策略比“启动时全量恢复所有房间”更简单，也更适合首版。
